
## 2024-06-05 - SQL Injection in Database Restore Endpoint via sql.raw
**Vulnerability:** An API route (`/api/settings/restore/route.ts`) accepted raw `.sql` file uploads and executed the lines directly using Drizzle's `sql.raw()`. The initial attempt to fix this by checking for table names and using regex to strip string literals before looking for semicolons was critically flawed, as the regex (`/'[^']*'/g`) failed to handle escaped quotes (`''`) or dollar-quoting (`$$`), allowing trivial bypasses and arbitrary code execution (e.g., `DROP TABLE`).
**Learning:** Custom parsing of raw SQL using Regex for security validation is fundamentally unsafe. String literals in SQL have complex rules (escaping, quoting styles like `$$`). When a real AST parser is unavailable and fallback manual parsing is necessary, it must use a robust character-by-character state machine rather than regex, and it must explicitly block advanced injection vectors like dollar-quoting and subqueries.
**Prevention:** Never use `sql.raw()` to execute untrusted upload content. If executing user-provided SQL is an unavoidable business requirement (like a backup restore feature), use an allowlist of target tables and employ a strict parser that thoroughly understands SQL string escaping semantics to prevent query stacking and subquery execution.
