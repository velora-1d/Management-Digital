## 2025-05-18 - [Critical] Authentication Bypass/IDOR in Profile API
**Vulnerability:** The `/api/profile/me` and `/api/profile/me/password` routes used a mock `getMe()` function that statically queried the database for the first non-deleted user (often the superadmin) instead of validating the authenticated session. This allowed any logged-in user to access and modify the profile and password of the first user in the database, leading to an Authentication Bypass and Insecure Direct Object Reference (IDOR).
**Learning:** Mock authentication functions that bypass token validation and directly query the database are extremely dangerous and can lead to severe security vulnerabilities like IDOR. All API endpoints must rely on proper authentication utilities (like `getAuthUser()` from `@/lib/rbac`) to retrieve the user's session and ID securely.
**Prevention:** Never use mock functions like `getMe()` in production code. Always use established authentication mechanisms (`getAuthUser()`) to fetch the authenticated user's context. Regularly audit API endpoints to ensure they validate sessions correctly rather than assuming authorization based on raw database queries.
