## 2024-06-19 - Timezone and Array Index Pitfalls When Refactoring Iterative Promises to SQL Group By
**Learning:** When refactoring iterative backend aggregations (like calculating monthly charts by looping over the last 6 months and making 2 queries per month inside `Promise.all`) into a single Drizzle ORM `groupBy` SQL query, naive array index referencing is a major pitfall. If standard chronological arrays are incorrectly assumed to be reverse-chronological, mapping bounds fail. Furthermore, replacing native Javascript `Date` timezone logic directly with database `AT TIME ZONE` extractions will cause off-by-one month drift at boundaries unless both sides use explicit timezone formats (like `Intl.DateTimeFormat` with `timeZone: 'Asia/Jakarta'`).
**Action:** When grouping aggregated dates into Javascript Maps, do not rely on local `m.start.getFullYear()`. Always synchronize the timezone extraction logic using `Intl.DateTimeFormat` on the JS side to explicitly match the SQL extraction, and always use explicit min/max finding or `length` checks rather than hardcoded array indices to determine query bounds.
